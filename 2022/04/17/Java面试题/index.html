Redis面试题

Redis单线程为什么这么快

  • 命令执行基于内存操作，一条命令在内存里操作的时间是纳秒级别
  • 命令执行是单线程操作，没有线程切换开销
  • 使用IO多路复用机制提升了Redis的IO利用率
  • 搞笑的数据存储结构（全局Hash表，跳表，压缩列表等）

Redis的Key过期了为什么内存没释放

Redis对于过期的Key处理一般有惰性删除和定时删除两种策略

  • 惰性删除：当读/写一个已经过期的Key时会触发惰性删除策略，判断Key是否过期，如果过期了就直接删除这个Key
  • 定时删除：Redis定期（默认每100ms）主动淘汰一批已经过期的Key，这里的一批只是部分过期的Key，所以还是会有部分Key已经过期但是没有被删除的情况

Redis的Key没设置过期时间为什么被Redis主动删除了

当Redis已用内存超过maxmemory限定时，触发内存淘汰策略
在Redis 4.0之前一共实现了6中内存淘汰策略，4.0之后，又增加了2中策略，总共8种：

  • 针对设置了过期时间的Key做处理
    • volatile-ttl：在筛选时，会针对设置了过期时间的键值对，根据过期时间额先后进行删除，越早过期的越先被删除
    • volatile-random：在设置了过期时间的的键值对中，进行随机删除
    • volatile-lru：在设置了过期时间的键值对中使用LRU算法筛选删除
    • volatile-lfu：在设置了过期时间的键值对中使用LFU算法筛选删除
  • 针对所有的Key做处理
    • allkeys-random：在所有键值对中随机删除
    • allkeys-lru：在所有键值对中使用LRU算法筛选删除
    • allkeys-lfu：在所有键值对中使用LRU算法筛选删除
  • 不处理
    • noeviction: 不会剔除任何数据，拒绝所有写入操作并返回客户端OOM错误信息

Redis淘汰Key的LRU算法和LFU算法

  • LRU（Least Recently Used，最近最少使用），淘汰距离当前时间最久的数据
  • LFU（Least Frequently Used，最不经常使用），淘汰截止当前时间访问次数最少的数据

Redis主从、哨兵、集群架构优缺点比较

  • 主从模式：一个主节点，多个从节点，客户端直接访问主节点。配置简单，单点故障。
  • 哨兵模式：一个主节点，多个从节点，增加一个哨兵集群，客户端通过哨兵访问主节点。配置复杂，依赖哨兵
  • 集群模式：多个主从合并组成集群

Redis持久化

  • RDB（Redis DataBase）：定时存储内存快照，文件体积小，恢复速度快，数据丢失较多
  • AOF（Append Only File）：记录除查询外的每一条操作指令，文件体积大，恢复数据慢，数据丢失较少

线上持久化设置

如果对性能要求较高，在Master最好不要做持久化，可以在某个Slave开启AOF备份数据，策略设置为每秒同步一次即可。

Redis集群为什么至少需要三个master节点

master节点的选举需要集群中大于半数master节点同意才能选举成功，如果有两个master节点，当其中一个挂了，是达不到选举新master的条件的

Redis集群为什么推荐奇数个节点

master节点的选举需要集群中大于半数master节点同意才能选举成功，奇数个节点可以在满足选举该条件的基础上节省一个节点。
比如三个master节点和四个master节点的集群相比，大家如果挂了一个master节点都能选举新master节点，如果都挂了两个master节点都没法选举新master节点了，所以奇数的master节点更多的是从节省机器资源角度出发来考虑的。

必考面试题

面向对象

对比面向过程，是两种不同的处理问题的角度
面向过程更注重事情的每一个步骤及顺序，面向对象更注重事情有哪些参与者（对象），及各自需要做什么。
面向过程比较直接高效，面向对象更易于复用，扩展和维护
面向对象中有三大特性：

  • 封装：对象的内部细节对外部透明，外部调用无需修改或者关心内部实现
  • 继承：继承基类的方法，并做出自己的改变或扩展
  • 多态：基于对象所属类的不同，外部对同一个方法的调用你，实际执行的逻辑不同

JDK, JRE和JVM

  • JVM：Java虚拟机
  • JRE：Java运行时环境，包含JVM和一些运行类库
  • JDK：Java开发工具包，包含JRE和Java工具（javac，java，jconsole）

== 和 equals

  • ==：基本数据类型是变量值，引用类型是堆中内存对象的地址
  • equals：只有引用类型有equals比较，object类中默认采用的是==比较，重写后按照重写的逻辑进行比较

final修饰符

修饰类：表示类不可被继承
修饰方法：表示方法不可被子类重写
修饰变量：表示变量一旦被赋值将不可以更该它的值（基本类型变量不可改，引用类型不可更改指向的对象）

为什么局部内部类和匿名内部类只能访问局部final变量

首先：内部类和外部类是处于同一个级别的，内部类不会因为定义在方法内中就随着方法的执行完毕就被销毁。
这样就会产生问题：当外部类的方法结束时，局部变量就会被销毁了，但是内部类对象可能还存在，就会产生内部类对象访问一个不存在的变量的问题。为了解决这个问题，就将局部变量复制了一份作为内部类的成员变量，这样当局部变量死亡后，内部类仍可以访问它，实际访问的是局部变量的副本。这样就好像延长了局部变量的生命周期。
将外部类的局部变量复制为内部类的成员变量时，必须保证这两个变量是一样的。为了保证这两个变量是一致的，java编译器规定我们必须将外部类的局部变量设置为final，保证变量在初始化后就无法修改，从而保证外部类和内部类的变量是一致的。

说一下ArrayList和LinkedList区别

首先，它们的底层数据结构不同，ArrayList底层是基于数组实现的，LinkedList底层是基于链表实现的。
由于底层数据结构不同，他们所适用的场景不同，ArrayList更适合随机查找，LinkedList更适合删除和添加。
另外ArrayList和LinkedList都实现了List接口，但是LinkedList还额外实现了Deque接口，所以LinkedList还可以当作队列来使用。

说一下HashMap的Put方法

将Key通过哈希算法算出哈希值，将哈希值与数组的长度-1相与得到数组下标值
如果数组下标位置元素为空，则将Key和Value封装为Entry或Node对象（JDK1.7 Entry/JDK1.8 Node），并放入该位置
如果下标不为空：

  1. 如果是JDK1.7，则先判断是否需要扩容，如果需要就进行扩容，如果不用扩容就生成Entry对象，使用头插法插入当前位置的链表中之后结束PUT方法
  2. 如果是JDK1.8，先判断当前位置上是红黑树Node还是链表Node
    1. 红黑树Node，则将Key和Value封装成一个红黑树Node并添加到红黑树中去，在添加过程中会判断是否存在这个Key，如果存在就更新该Key的Value
    2. 链表Node，则将Key和Value封装成一个链表Node并添加到链表中，添加链表使用的是尾插法，会遍历链表中的Node，如果链表中存在当前的Key，则更新该Key的Value。
    3. 更新之后会看当前链表的长度是否大于等于8，是的话执行链表转红黑树的方法
    4. 最后判断是否需要进行扩容，是的话进行扩容，否的话结束PUT方法

说一下ThreadLocal

ThreadLocal是Java中所提供的线程本地存储机制，可以利用该机制将数据缓存在某个线程内部，该线程可以在任意时刻，任意方法中获取缓存数据。
ThreadLocal底层是通过ThreadLocalMap来实现的，每个Thread对象中存在一个ThreadLocalMap，Map的Key为ThreadLocal对象，Map的Value为需要缓存的值。
如果在线程池中使用ThreadLocal会造成内存泄漏，因为线程池中的线程不会被回收，线程对象通过强引用指向ThreadLocalMap，ThreadLocalMap通过强引用指向Entry对象，线程不会被回收则Entry对象也不会被回收，从而造成内存泄漏。解决办法是在使用完ThreadLocal对象之后，手动调用ThreadLocal的Remove方法，手动清除Entry对象。
ThreadLocal经典的应用场景就是数据库的连接管理。
